Component Pattern (inferred from Testimonials Slider)

Every custom block follows the same architecture:

blocks/[name]/ ├── block.json ← Registration, attributes, keywords ├── editor.js ← Gutenberg sidebar controls + placeholder preview ├── render.php ← Server-side WP_Query + HTML output └── (no view.js) ← Front-end JS lives in assets/js/ and is conditionally enqueued inc/[name]-block.php ← wp_register_block_type() call assets/css/[name].css ← Standalone stylesheet, enqueued by block or conditionally assets/js/[name].js ← Front-end interactivity (slider, filter, etc.)

Key observations:

  • Data comes from CPTs via WP_Query — never stored in block attributes (except config like speed, group ID).
  • Carbon Fields stores structured metadata; taxonomy terms handle categorisation.
  • CSS uses theme.json design tokens: var(--wp--preset--*) for colours, spacing, font sizes.
  • Blocks are conditionally enqueued (see project-enqueue.php pattern).
  • All front-end HTML is server-rendered PHP — no React hydration on the front-end.

2. Staging Environment Setup

Recommended Hosting: Flywheel or Cloudways

Both offer one-click WordPress staging with SSL. Flywheel pairs naturally with Local by Flywheel for push/pull. Cloudways is more flexible if you want DigitalOcean/Vultr under the hood.

Option A — Flywheel (Simplest)

  • Create a Flywheel account and new site (e.g. vpe-staging.flywheelsites.com)
  • In Local, right-click the site → Push to Flywheel (uploads files + DB)
  • Enable free SSL on the Flywheel dashboard
  • Password-protect the staging URL via Flywheel's built-in option

Option B — Cloudways / Generic Host

  • Provision a server + WordPress application
  • Install Spectra One parent theme + activate child theme
  • Run composer install in the child theme directory
  • Import the Local database via WP Migrate or phpMyAdmin
  • Search-replace spectra.local → staging URL in the DB
  • Enable SSL (Let's Encrypt / Cloudflare)

Git-Based Deployment Workflow

The theme is already versioned on GitHub. Set up a deploy pipeline so every push to main auto-deploys to staging:

Lightweight: GitHub + Deploy Hook

# On the staging server, create a deploy script:
# /home/deploy/deploy-theme.sh

cd /var/www/html/wp-content/themes/spectra-child
git pull origin main
composer install --no-dev --optimize-autoloader

Trigger this via a GitHub Actions workflow or a webhook on push to main.

GitHub Actions Example

# .github/workflows/deploy-staging.yml
name: Deploy to Staging
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.STAGING_HOST }}
          username: ${{ secrets.STAGING_USER }}
          key: ${{ secrets.STAGING_KEY }}
          script: |
            cd /path/to/themes/spectra-child
            git pull origin main
            composer install --no-dev --optimize-autoloader

  • Document any manual WP Admin steps (e.g., FSE template edits, menu configs) in a SETUP.md

3. Component Standardisation Strategy

Established Pattern

The codebase already follows a consistent architecture. Codify it as the standard for all future components:

Block Component Anatomy

FileResponsibility
blocks/[name]/block.jsonRegistration: name, attributes (config only — no content data), keywords, editor script, render path
blocks/[name]/editor.jsGutenberg placeholder UI + sidebar controls (InspectorControls). Fetches live count from REST for preview.
blocks/[name]/render.phpServer-side WP_Query, Carbon Fields reads, full HTML output. All data fetched at render time.
inc/[name]-block.phpwp_register_block_type() call, conditional enqueue logic if needed.
assets/css/[name].cssComponent-scoped styles using BEM naming and var(--wp--preset--*) tokens.
assets/js/[name].jsFront-end interactivity (sliders, filters). Vanilla JS — no jQuery, no React.

CSS Architecture

  • Global styles: style.css — hero, trust strip, page-level layout, button system
  • Component styles: assets/css/[name].css — scoped to one block, conditionally loaded
  • Design tokens: Always use var(--wp--preset--color--*), var(--wp--preset--spacing--*), var(--wp--preset--font-size--*) instead of hard-coded values
  • Accessibility: Every interactive element needs :focus-visible styles. Use prefers-reduced-motion media query for transitions.

7. Risks & Gaps

Risk wpautop HTML Mangling

WordPress's wpautop filter inserts <p> and <br> tags inside shortcode output rendered through the FSE Shortcode block. This has already caused duplicate elements and broken layouts in the featured project hero. Mitigation: keep all HTML inside <a> tags on a single output line. Consider building a small utility function that strips wpautop artifacts from shortcode returns.

Risk theme.json Not Extended

The current theme.json only registers custom templates. It does not define the colour palette, typography, or spacing scales — these are inherited from Spectra One. If the parent theme updates and changes defaults, child theme styling could break. Mitigation: extend theme.json with explicit settings.color.palette, settings.typography, and settings.spacing blocks to lock in the VPE design system.

Risk Carbon Fields Dependency

Carbon Fields is loaded via Composer's vendor/autoload.php directly in the theme. If the vendor/ directory is missing on deploy (it's gitignored), the site will fatal error. Mitigation: ensure every deploy script runs composer install --no-dev. Add a safety check in functions.php: if (!file_exists(__DIR__ . '/vendor/autoload.php')) return;

Risk No Automated Testing

There are no unit tests or integration tests. Shortcode output, REST API responses, and block rendering are tested manually. Mitigation: at minimum, add a health-check endpoint or WP-CLI script that verifies CPTs exist, taxonomy terms are populated, and Carbon Fields is loaded. Consider adding PHPUnit tests for shortcode functions before launch.

Resolved Large functions.php

Previously: At 855 lines, functions.php handled CPT registration, taxonomy registration, Carbon Fields setup, and 8+ shortcodes. Resolution: extracted into three dedicated files — inc/post-types.php (CPTs, taxonomies, term populators), inc/carbon-fields-setup.php (Carbon Fields boot + field registrations), and inc/video-project-shortcodes.php (helper + all shortcodes). functions.php is now 54 lines and only bootstraps includes + small utilities.

Risk No Content Staging Strategy

FSE template edits, menu configurations, and widget/block content live in the database — not in Git. If the staging DB diverges from local, template changes can be lost during sync. Mitigation: designate staging as the source of truth for content. Only push theme code via Git. Pull DB from staging to local when needed, never overwrite staging DB from local without coordination.

Gap Missing SEO & Analytics

No SEO plugin, sitemap, or analytics tracking is configured. Action: install Yoast SEO or RankMath during Phase 6. Add Google Analytics / GA4 via a lightweight plugin or theme.json custom HTML block in the header template part.

Gap Missing Performance Optimisation

No caching plugin, no image optimisation pipeline. Vimeo hero video could cause LCP issues on slow connections. Action: install a caching plugin (WP Super Cache or LiteSpeed Cache), add image optimisation (ShortPixel or Imagify), and add a poster image to the hero video for fast LCP.

8. QA & Launch Checklist

Pre-Launch (Staging)

  • All pages render without PHP warnings or errors
  • Responsive: tested at 375px, 768px, 1024px, 1440px
  • Keyboard navigation works on all interactive elements
  • Screen reader tested (VoiceOver / NVDA) on key pages
  • Colour contrast passes WCAG AA (4.5:1 for body text)
  • All links point to correct destinations (no broken links)
  • Contact form sends emails correctly
  • Video embeds load and play on mobile + desktop
  • Project filter works across all taxonomy combinations
  • Testimonials slider navigates correctly with 1, 2, and 5+ cards
  • Logo carousel renders with correct logo heights
  • Lighthouse score: Performance ≥ 80, Accessibility ≥ 90, SEO ≥ 90
  • Open Graph / social preview images set for key pages
  • 404 page is styled and helpful
  • XML sitemap is generated and submitted to Google Search Console

Launch Day

  • DNS updated to point to production server
  • SSL certificate verified on production
  • Search-replace staging URL → production URL in DB
  • Remove staging password protection on production
  • Flush all caches (object, page, CDN)
  • Verify all pages on production URL
  • Submit production sitemap to Google Search Console
  • Set up uptime monitoring (UptimeRobot or similar)
  • Tag Git release: git tag v1.0.0 && git push --tags
